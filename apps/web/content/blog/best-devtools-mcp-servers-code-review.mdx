## GitHub MCP

Published and maintained by GitHub at github.com/github/github-mcp-server, this server is the starting point for any AI-assisted PR review workflow. It exposes GitHub's REST and GraphQL APIs as typed MCP tools, giving an agent structured access to PR diffs, file contents, review threads, CI status, and the ability to post review comments, all without leaving the conversation. Open source, MIT license, with commits from the GitHub team landing weekly.

The tools that matter for code review are `get_pull_request` (returns the full diff plus metadata: base branch, CI check status, reviewer assignments), `get_pull_request_files` (lists each changed file with its patch), `create_review_comment` (posts an inline comment on a specific line of the diff), and `submit_pull_request_review` (submits a full review with APPROVE, COMMENT, or REQUEST_CHANGES status). Those four together let an agent fetch a PR, reason over every changed file, and submit a structured first-pass review with specific line comments, all in one agent loop.

This is the right server if your team lives on GitHub and you want an agent to either summarize what a PR changes or leave actual review comments that show up in the pull request thread. It pairs cleanly with every other server in this list: the other servers do the analysis, GitHub MCP handles the delivery. Auth is a fine-grained personal access token scoped to specific repos and operations, or a GitHub Apps installation for org-wide rollouts.

Skip it if your team uses GitLab, Bitbucket, or Azure DevOps. GitHub MCP covers GitHub only. For multi-provider teams, GitKraken GK-CLI (covered in the [Repo Intelligence listicle](/blog/best-devtools-mcp-servers-repo-intelligence)) provides cross-platform PR access, though with less review-action depth.

12,000+ stars on github.com/github/github-mcp-server. MCPFind directory star count is pending enrichment, but maintenance evidence is strong: open issues receive responses within days. Apache 2.0 license.

## rsdoctor

From the Rspack ecosystem team (github.com/web-infra-dev/rsdoctor), rsdoctor is a build analyzer and code quality diagnostic tool for Rspack and webpack projects. With 1,144 stars on MCPFind, it's the highest-adoption specialized analysis tool in the HEALTHY devtools tier. The MCP server wraps rsdoctor's analysis capabilities so an agent can query bundle composition, module dependency trees, duplicate package detection, and build performance metrics without running a separate CLI. Last push March 24, 2026.

The tools give an agent access to build-time signals that rarely show up in code review today. `analyze_bundle` runs a full bundle analysis and returns a structured report covering module sizes, chunk composition, and tree-shaking effectiveness. `find_duplicate_packages` identifies the same package appearing at multiple versions in the dependency graph, a common source of bundle bloat that gets introduced silently when a new library brings its own transitive dependencies. `get_module_graph` returns the full dependency graph for a specific entry point, which is useful when a PR adds a new import and you want to know what it pulls in. `analyze_build_performance` surfaces which loaders and plugins are eating the most build time, helpful for catching slow build additions before they compound.

Reach for this if your team ships web applications built on Rspack or webpack and you want an agent to flag bundle quality issues during PR review. The specific case it covers well: a PR adds a new library and the reviewer wants to know the size cost, whether it duplicates something already in the bundle, and how it affects the dependency graph. That question usually goes unanswered in code review today because answering it requires running the analyzer manually.

It won't help with source code quality analysis, standards enforcement, or security scanning. rsdoctor reads the build output, not the source; it can't tell you whether the code style is correct or whether a function is too complex. It also only covers Rspack and webpack projects. Vite, Turbopack, and esbuild-based builds aren't supported.

1,144 stars, last push March 24, 2026. Maintained by the Rspack team (ByteDance open-source). MIT license. Open issues see active responses.

## Packmind MCP

Packmind (247 stars on MCPFind, at ai.packmind/mcp-server) occupies a niche that no linter fills: applying your team's documented coding standards to a PR diff and returning specific violations with suggested fixes. Where ESLint enforces JavaScript conventions and TypeScript checks types, Packmind enforces the patterns your team decided on, the ones that live in Confluence pages, architecture decision records, or Notion docs that reviewers are supposed to remember but don't consistently apply. The server reads from a Packmind workspace where those guidelines are stored, then runs them against a diff. Last push March 24, 2026.

`packmind_review_pr` takes a repo and PR identifier and returns the list of guideline violations found in the diff, organized by file and line number. `packmind_get_guidelines` pulls your team's full rule set as structured data, useful when the agent needs to explain why a pattern is required rather than just flag the violation. `packmind_suggest_fix` generates a proposed code change for a specific violation that the agent can either apply directly or surface as a review comment through GitHub MCP. The end-to-end flow: GitHub MCP fetches the PR diff, Packmind applies the rules, GitHub MCP posts the resulting comments. A human reviewer sees structured feedback before they even open the file.

This server is the right pick if your team has already invested in documenting standards and wants to automate the enforcement step. Teams that hold regular architecture reviews, maintain an engineering handbook, or have style guides that pre-date a linter are the natural fit. The return on setup is highest when the gap between documented standards and what actually lands in PRs is already a source of friction in reviews.

Packmind doesn't ship its own rule set. It applies rules you defined in your workspace. Teams without written standards will find nothing to enforce. For that situation, generic linters or a static analysis tool is a better starting point. Packmind is the automation layer on top of standards work that's already done.

247 stars, last push March 24, 2026. Auth requires a Packmind API key generated from your workspace settings. Commercial product with a free trial and per-seat pricing for teams. Issues see responses from the Packmind team.

## Code Pathfinder

From the team at codepathfinder.dev (113 stars on MCPFind), Code Pathfinder handles the question a reviewer needs to answer before approving anything: "what breaks if this changes." It parses source code at the AST level and answers structural queries precisely, not by scanning text but by understanding the code's shape. Find every call site of a function. List every class that implements an interface. Map which modules depend on a file being modified. Those are the questions that matter when a PR touches a shared utility or changes a public API signature. Last push March 24, 2026.

`pathfinder_find_callers` returns every call site for a given function across the codebase, the tool to reach for before approving a change to any function that other code depends on. `pathfinder_find_symbol` locates function definitions, class declarations, and variable assignments by name. `pathfinder_find_implementations` lists every concrete class or function that implements an abstract interface, which matters when a PR adds a field to an interface and every implementation site needs updating. `pathfinder_dependency_graph` returns a module-level dependency map showing which files import which others, useful for estimating the blast radius of a structural change.

The specific reviewer workflow that fits: a PR modifies a utility function used in multiple places. Before approving, you want to know whether all call sites handle the new behavior correctly. Code Pathfinder lists every caller with file and line number. The reviewer (or the agent) checks each site. That's a mechanical task that currently requires grep or IDE search; Code Pathfinder answers it as a typed query.

There's nothing here for PR state, review comment delivery, or standards enforcement. Code Pathfinder's scope is the local codebase. It works on code sitting on disk and has no GitHub connection. Pair it with GitHub MCP when you need both impact analysis and the ability to post review comments in one conversation.

113 stars, last push March 24, 2026. Runs locally, no credentials needed. Open source. Documented language support includes TypeScript, JavaScript, Python, and Go.

## Sonatype Dependency Management MCP

Published by Sonatype at github.com/sonatype-nexus-community/mcp-sonatype (69 stars), this server brings Sonatype's component intelligence into an AI agent's code review workflow. Sonatype built the OSS Index, a free vulnerability database covering millions of open-source components across Maven, npm, PyPI, and other ecosystems. The MCP server exposes that database as typed tools, so an agent can check every new dependency a PR introduces against known vulnerability records and license policy. Last push January 14, 2026.

`check_component` takes a package identifier (name plus version, in the ecosystem's format) and returns the vulnerability report from OSS Index: CVE identifiers, severity scores, affected version ranges, and whether the installed version is patched. `analyze_manifest` accepts a package manifest file (package.json, pom.xml, requirements.txt) and returns a full report for all listed dependencies at once, which is the tool to run against the diff when a PR modifies a dependency file. For teams with Sonatype Lifecycle (the commercial product), `policy_evaluation` runs the component list against your organization's defined license and security policies and returns pass/fail per component.

Security-aware teams reviewing PRs that add or update dependencies should have this in the stack. The pattern that works well: GitHub MCP extracts the changed manifest file from the PR diff, Sonatype checks every new or updated package for CVEs, the agent reports any findings before the PR merges. That catches the scenario where a well-intentioned dependency update quietly brings in a transitive package with a known exploit.

Teams working exclusively with private or internal packages won't get coverage from OSS Index, which tracks public open-source components. The server is also read-only by design; it reports findings but doesn't automate remediation. Dependency upgrade suggestions need to come from the development workflow, not this server.

69 stars, last push January 14, 2026. OSS Index queries are free with no API key required; Sonatype IQ Server integration requires a commercial license. Open source under Apache 2.0. The January 2026 last push is the most recent in the directory snapshot; the underlying OSS Index database updates continuously.

## CodeScene MCP

CodeScene (32 stars, at github.com/codescene-oss/codescene-vscode, published by Empear) takes a different angle on code quality: instead of checking style or vulnerabilities, it identifies where the code is most likely to cause problems based on change history and complexity analysis. CodeScene's core insight is that technical debt accumulates in predictable hotspots. Files changed frequently with high cyclomatic complexity are the ones that slow teams down and introduce bugs. The MCP server surfaces those signals as typed queries so an agent can incorporate CodeScene's health data into a code review. Last push March 24, 2026.

`analyze_file_health` takes a file path and returns CodeScene's code health score along with the specific issues driving the score: deeply nested logic, large functions, missing function abstractions, and high coupling with other modules. `get_hotspots` returns the files in the codebase ranked by change frequency combined with complexity, the combination that predicts where defects are most likely to cluster. `get_refactoring_targets` surfaces specific functions or modules that CodeScene identifies as high-priority candidates for cleanup based on health trends over time. For teams running CodeScene's commercial platform, `get_delivery_risk` returns a risk assessment for a set of changed files based on their historical defect correlation.

The code review use case that fits: a PR touches a file already flagged as a hotspot, or introduces changes that push a module's complexity score past a threshold. CodeScene lets the agent note that context in the review, so the human reviewer sees the quality signal alongside the diff itself. That's meaningfully different from a linter flag; it's historical context about how much trouble that file has caused before.

CodeScene doesn't catch bugs or enforce style. It surfaces complexity and health trends. Teams without an existing CodeScene account need to set one up before the MCP server has data to return; there's nothing to query against a fresh codebase with no analysis history. The platform is commercial; the free tier supports small teams.

32 stars, last push March 24, 2026. Auth uses a CodeScene API key. Commercial product with a free tier. Issues see responses from the Empear team.

## How to Choose: Decision Framework

Start with what you actually want the agent to do during a review. Posting structured comments on a PR diff is a GitHub MCP job; that's the only server in this list that can write back to the PR thread. Every other server does analysis and returns findings, but delivery of those findings as actual GitHub review comments requires GitHub MCP in the chain. If you want a summary in chat rather than comments posted to the PR, any server's output works without GitHub MCP.

For teams that want to catch rule violations specific to their engineering standards, Packmind is the right layer. It enforces the documented patterns that generic linters don't cover. The prerequisite is that those standards exist in written form. If the rule isn't in the Packmind workspace, it won't be checked. Teams starting from scratch on standards documentation are better off establishing the rules first and wiring in Packmind after.

Dependency security belongs in every review stack where PRs can touch package manifests. Sonatype's OSS Index is free and covers Maven, npm, PyPI, and NuGet. Adding Sonatype to the chain means a dependency addition that imports a CVE gets caught automatically, before a human reviewer has to spot it manually. That's a low-setup, high-value addition.

For larger codebases where the question "what else does this change affect" is genuinely hard to answer quickly, Code Pathfinder handles the AST-level impact query. Run it against functions or interfaces being modified to get a precise list of call sites and implementations. CodeScene adds the longitudinal dimension: which files have historically been the most problematic, and does this PR touch any of them. The two are complementary rather than overlapping.

## Frequently Asked Questions

### Can MCP servers automate the entire code review process?

Not entirely, but they cover the mechanical parts well. MCP servers can check a diff against documented coding standards, flag dependency vulnerabilities, identify high-complexity hotspots, and generate structured review comments. The judgment calls that require business context or architectural intuition still need a human. The best use case is catching rule-based violations before human reviewers see the PR.

### Do code review MCP servers work with private repositories?

Yes. Servers that connect to GitHub or similar hosts use token-based auth that inherits exactly the permissions you grant. A fine-grained GitHub PAT scoped to specific private repositories is the standard approach. Servers that run locally against a checked-out codebase (CodeScene, Code Pathfinder) never send code to an external service at all.

### Which code review MCP server is best for enforcing team-specific coding standards?

Packmind is built for exactly this. It reads coding guidelines your team has already written down, then applies those rules to a PR diff and returns specific violations with suggested fixes. Generic linters enforce language conventions; Packmind enforces your team's documented patterns. If your standards aren't written down yet, start with Packmind's workspace setup before wiring in the MCP server.

### Can I run multiple code review MCP servers at the same time?

Yes. Each server handles a different concern, and they don't conflict. A common setup pairs GitHub MCP (to fetch PR context) with Packmind (to apply standards) and Sonatype (to check new dependencies for vulnerabilities). All three can run in parallel within the same MCP client config. The agent combines the results in one response.

### Are there free code review MCP servers, or are they all commercial products?

Both exist. GitHub MCP and Code Pathfinder are open source and free to self-host. Packmind and CodeScene are commercial products with free tiers for small teams. Sonatype's OSS Index (which the Dependency Management MCP uses) is free for open-source components; the full IQ Server product requires a license. Check each vendor's pricing page for current team and seat limits.
